Option A — SQLCipher (whole-DB AES-256-CBC)
  Key = HKDF-SHA3-256(wallet_priv, "QTCL-DB-ENC-v1"). Every page encrypted
  transparently — zero schema changes, no per-row overhead. Works in Python
  via pysqlcipher3. Rotation = PRAGMA rekey.

  Option B — Field-level ChaCha20-Poly1305
  Encrypt only sensitive columns: density_matrix_hex, hyp_signature_hex,
  sender_wallet_addr. Columns like w_state_fidelity and block_height stay
  plaintext for SQL WHERE/ORDER BY. Key = same wallet-derived HKDF output.
  Each ciphertext is nonce(12B) || tag(16B) || ct. Decryption happens in
  Python before use. This lets you query indexes without decrypting.

  Option C — GeodesicLWE at rest (full HypΓ)
  Run HypGammaCipher.encrypt(dm_bytes, oracle_pubkey) from hyp_lwe.py before
  writing to oracle_dm_cache.density_matrix_hex. The lattice error vector
  becomes the encoding; decryption requires HypGammaCipher.decrypt(ct,
  oracle_privkey). Attacker who extracts the SQLite file cannot recover any DM
   without the private key. Most secure; roughly 2–4× storage overhead due to
  lattice ciphertext expansion.
 Option D — Application-level encrypt-before-ingest
  Before the qtcl_gossipIngest call in _sync_oracle_dm_to_neon, encrypt
  density_matrix_hex with AES-256-GCM (key = per-epoch wallet-derived key,
  IV = sha3_256(key || block_height)[:12]). Store ciphertext + auth_tag in
  the JSON payload. Server stores opaque blob; only nodes with matching
  epoch key can read it. Key epoch = every 1000 blocks.

  Option E — HypΓ commitment scheme (zero-knowledge)
  Store commit = sha3_256(dm_hex || blinding_r) in Neon alongside
  r_encrypted = AES(r, wallet_key). The DM itself never leaves the client in
   plaintext. Any node can verify sha3_256(dm_hex_disclosed || r) == commit
  if you later choose to disclose. Neon only ever sees commitments — full ZK
   from the DB's perspective.
